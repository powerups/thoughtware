Admirable Qualities of a Tech Company

2013/11/25

This list represents a set of attributes which professional acquaintances and I, after working at many technology companies, have collectively formed. Our experience includes working everywhere from Fortune 50′s to Bay-Area Startups, from fresh codebases to 30-year-old relics.

This represents an idealized working environment for not only us but what I can only assume to be very many software developers. While many companies would consider compliance of this list an impossibility, getting as close to this ideal as possible will ensure many developers, myself included, will want to work for you.

  • Should contribute to the open source community
  • Should allow developers to use their own development machine
    • Not being able to do so is usually a sign of a too-tightly coupled development environment…
  • Should not require proprietary communication protocols
    • Does your Microsoft Exchange server not work with Linux or OS X?
    • Does your documentation require Microsoft Office or Pages for Mac?
  • Should allow developers to take unpaid vacation
    • Is it that important that I only get the allotted X days of vacation per year?
  • Should not have a codebase full of spaghetti code
    • Consider switching to a Service Oriented Architecture, it helps ;)
  • Should prioritize time to refactor / fix technical debt and other things which don’t directly generate revenue
    • If technical debt is ever added for some reason, it should be documented and removed when possible
  • Should not use a “custom framework” or incorrectly implement an open source framework
    • I should be able to solve most code-related questions I have using Stack Overflow.
  • Should not be a “marketing-driven” company
  • Should not have arbitrary deadlines
    • “We gotta get this feature out by Tuesday, that’s when we’re sending out the email blast!”
  • Should throw away the Minimum Viable Product
    • Trust me, it sucks, you’ll be happier when it’s gone.
  • Should not require employees to be in the office all day, e.g., can work at a coffee shop or from home
  • Should promote based on merit, not solely based on how long a developer has stuck around
  • Should first consider feedback from developers before making large developer-affecting decisions
  • Should assign tasks to developers based on what the developers would like to do
